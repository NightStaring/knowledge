## 1. 什么是消息队列

**消息队列（Message Queue, MQ）** 是一种进程间通信或系统间通信的方式，通过**异步**传递消息来实现解耦。生产者将消息发送到队列，消费者从队列中拉取或由队列推送消息进行处理。

### 1.1 为什么要用消息队列

| 场景 | 说明 |
|------|------|
| **解耦** | 生产者和消费者不直接依赖，一方变化不影响另一方 |
| **异步** | 非核心流程可以异步处理，减少主流程耗时 |
| **削峰填谷** | 高流量时消息暂存队列，下游按能力消费，防止系统被冲垮 |
| **流量控制** | 通过限制消费速率保护后端服务 |
| **广播** | 一条消息分发给多个消费者（Pub/Sub） |
| **最终一致性** | 配合可靠投递实现分布式系统的事务最终一致性 |

### 1.2 典型应用场景

- **任务异步处理**：用户注册后发送邮件/短信
- **日志收集**：多个服务产生的日志统一发送到日志系统
- **事件驱动架构**：订单创建 → 库存扣减 → 积分增加 → 通知推送
- **数据同步**：跨服务/跨数据库的数据最终一致同步
- **流式处理**：实时计算、ETL 管道

## 2. 核心概念

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Producer │────>│  Broker  │────>│ Consumer │
│ (生产者)  │     │  (代理)   │     │ (消费者)  │
└──────────┘     └──────────┘     └──────────┘
                       │
                  ┌────┴────┐
                  │  Queue  │
                  │ (队列)   │
                  └─────────┘
```

### 2.1 Producer（生产者）
发送消息的一方。生产者将消息发布到 Broker 上的某个 Queue 或 Topic。

### 2.2 Consumer（消费者）
接收并处理消息的一方。消费者订阅队列或主题，消费其中的消息。

### 2.3 Broker（代理 / 服务器）
消息队列的服务器端，负责接收、存储、路由和投递消息。

### 2.4 Queue（队列）
消息的存储容器，遵循 FIFO（先进先出）原则。消息被消费后通常从队列中删除。

### 2.5 Topic（主题）
在发布/订阅模型中，消息按主题分类。生产者发送到主题，消费者订阅主题。

### 2.6 Exchange（交换器）
在 AMQP 模型中（如 RabbitMQ），生产者不直接发送到队列，而是发送到 Exchange，由 Exchange 根据路由规则分发到队列。

### 2.7 Message（消息）
传输的基本单元，通常包含：
- **Headers** — 元数据（路由键、消息 ID、时间戳、优先级等）
- **Body** — 实际数据（通常是 JSON、XML 或二进制）
- **Properties** — 属性（持久化标记、TTL、投递模式等）

### 2.8 Consumer Group（消费者组）
一组消费者共同消费一个主题/队列，每条消息只被组内的一个消费者处理（Kafka 的核心概念）。

## 3. 消息模型

### 3.1 点对点模型（Point-to-Point / Queue Model）

```
Producer ──→ [Queue] ──→ Consumer A
                    └──→ Consumer B (竞争消费)
```

- 每条消息只被一个消费者消费
- 消息消费后从队列中删除
- 典型实现：RabbitMQ Queue、ActiveMQ Queue

**适用场景**：任务分发、负载均衡

### 3.2 发布/订阅模型（Publish-Subscribe / Topic Model）

```
Producer ──→ [Topic] ──→ Subscriber A
                   ├──→ Subscriber B
                   └──→ Subscriber C
```

- 一条消息可被多个消费者接收
- 消费者各自独立消费
- 典型实现：Kafka Topic、RabbitMQ Fanout Exchange

**适用场景**：事件广播、数据分发、日志收集

### 3.3 混合模型

现代消息队列通常同时支持两种模型：
- **RabbitMQ**：通过 Exchange + Binding 实现灵活的路由
- **Kafka**：通过 Consumer Group 实现单播，Group 间实现广播

## 4. 消息协议

### 4.1 AMQP（Advanced Message Queuing Protocol）
- **特点**：功能丰富、二进制协议、标准化（ISO 19464）
- **版本**：0-9-1（RabbitMQ 主要使用）、1.0（更现代但兼容性不同）
- **核心**：Exchange → Binding → Queue 的路由模型
- **代表**：RabbitMQ
- **适用**：企业级应用、复杂路由场景

### 4.2 MQTT（Message Queuing Telemetry Transport）
- **特点**：轻量级、低带宽、低功耗、发布/订阅
- **QoS 级别**：0（至多一次）、1（至少一次）、2（恰好一次）
- **代表**：EMQX、Mosquitto
- **适用**：IoT、移动设备、传感器网络

### 4.3 STOMP（Simple Text Oriented Messaging Protocol）
- **特点**：基于文本、简单易用、类似 HTTP
- **代表**：ActiveMQ、RabbitMQ（通过插件支持）
- **适用**：多语言互操作、WebSocket 场景

### 4.4 Kafka 自定义协议
- **特点**：基于 TCP 的二进制协议、高性能
- **代表**：Apache Kafka、Confluent
- **适用**：高吞吐日志、事件流、数据管道

## 5. 投递保证

消息投递的三种保证级别：

| 级别 | 说明 | 可能问题 | 典型实现 |
|------|------|----------|----------|
| **At-most-once**（至多一次） | 消息最多投递一次，可能丢失 | 丢消息 | 非持久化、不确认 |
| **At-least-once**（至少一次） | 消息至少投递一次，不会丢失但可能重复 | 重复消费 | 确认机制 + 重试 |
| **Exactly-once**（恰好一次） | 消息不会丢也不会重复 | 性能开销大 | 事务 + 幂等消费 |

### 5.1 实现机制

**生产者端：**
- **Confirm / Ack** — 生产者发送后等待 Broker 确认
- **事务** — 批量发送后提交，性能较差
- **重试** — 发送失败自动重试（需配合幂等）

**Broker 端：**
- **持久化** — 消息写入磁盘
- **副本** — 多副本冗余（Kafka 的 ISR 机制）
- **同步刷盘** — 写入后同步 fsync

**消费者端：**
- **手动 Ack** — 处理完后再确认，失败则重新投递
- **幂等消费** — 消费端去重（唯一 ID + 状态表 / 业务键）
- **死信队列** — 多次重试失败的消息转入死信队列

## 6. 核心特性

### 6.1 持久化（Persistence）
- 消息保存到磁盘，Broker 重启后不丢失
- 权衡：持久化会降低吞吐，可通过异步刷盘优化

### 6.2 顺序保证（Ordering）
- 单个分区/队列内消息有序
- 多分区/多队列场景需要业务端处理顺序
- 关键权衡：**严格顺序 = 低并发**，需在一致性和性能间平衡

### 6.3 重试机制（Retry）
- 消费失败自动重试
- 可配置重试次数、间隔、退避策略
- 超过重试上限进入死信队列（DLQ）

### 6.4 死信队列（Dead Letter Queue, DLQ）
- 存储无法正常处理的消息
- 用途：排查问题、人工处理、延迟重试
- 通常需要配合告警监控

### 6.5 延迟消息（Delayed Message）
- 消息在指定延迟后才投递到消费者
- 用途：订单超时取消、定时任务
- 实现方式：插件（RabbitMQ delayed message）、时间轮、Kafka 时间索引

### 6.6 消息回溯（Message Rewind / Offset Reset）
- 消费者可以从之前的某个位置重新消费
- Kafka 通过 Offset 机制天然支持
- RabbitMQ 需要插件或单独设计

### 6.7 流量控制（Flow Control）
- **Consumer 侧**：限速消费、预取限制（QoS prefetch）
- **Producer 侧**：背压（Backpressure）、限流

### 6.8 事务消息
- 保证生产者的本地事务和消息发送的原子性
- RocketMQ 的事务消息是典型实现
- 适用：分布式事务的最终一致性方案

## 7. 常见模式

### 7.1 工作队列（Work Queue / Task Queue）

```
                  ┌── Worker 1
Producer ──→ Queue ── Worker 2
                  └── Worker 3
```

- 多个消费者竞争消费，任务负载均衡
- 适用于耗时任务的分布式处理

### 7.2 发布/订阅（Pub/Sub）

```
                  ┌── Consumer A (Group-1)
Producer ──→ Topic ── Consumer B (Group-1)
                  └── Consumer C (Group-2)
```

- 一条消息广播给所有订阅者
- 适用于事件通知、数据同步

### 7.3 路由模式（Routing）

```
Producer ──→ Exchange ──binding key "error"──→ Error Queue
                   └──binding key "warn" ──→ Warn Queue
                   └──binding key "info"  ──→ Info Queue
```

- 根据路由键选择性分发消息
- RabbitMQ Direct Exchange 的典型用法

### 7.4 主题模式（Topic）

```
Producer ──→ Topic Exchange ──binding "*.error"──→ Error Queue
                        └──binding "#.log"  ──→ Log Queue
```

- 支持通配符路由（`*` 匹配一级，`#` 匹配多级）
- 灵活的分发规则

### 7.5 请求/应答（Request-Reply）

```
Client ──→ Request Queue ──→ Server
   ←── Reply Queue ←──────
```

- 用于 RPC 调用
- 通过 Correlation ID 关联请求和响应

### 7.6 死信 + 延迟重试

```
                 ┌── 重试队列 ──→ 重新入队
Producer ──→ 主队列 ── 消费失败 ──→ DLQ
                                  └── 告警/人工处理
```

- 消费失败后先重试，超限转入死信
- 可与延迟队列配合实现间隔重试

### 7.7 事件溯源 / Event Sourcing

```
Service A ──→ Event Log ──→ Service B (更新读模型)
                       └──→ Service C (触发后续流程)
                       └──→ 数据分析 (离线消费)
```

- 以事件为中心，所有状态变化作为不可变事件存储
- Kafka 是这类模式的最佳载体

## 8. 选型对比

| 维度 | RabbitMQ | Apache Kafka | RocketMQ | ActiveMQ |
|------|----------|-------------|----------|----------|
| **定位** | 通用消息代理 | 分布式流平台 | 低延迟消息中间件 | 传统 JMS 实现 |
| **协议** | AMQP 0-9-1 / MQTT / STOMP | 自定义 TCP 协议 | 自定义协议 | OpenWire / STOMP / AMQP |
| **消息模型** | Exchange + Queue | Topic + Partition | Topic + Queue | Queue + Topic |
| **顺序保证** | 单队列有序 | 单分区有序 | 单队列有序 | 单队列有序 |
| **持久化** | 内存/磁盘 | 磁盘（追加写） | 磁盘 | 磁盘 |
| **吞吐量** | 万级/秒 | 百万级/秒 | 十万级/秒 | 万级/秒 |
| **延迟** | 微秒级 | 毫秒级 | 毫秒级 | 微秒级 |
| **消息回溯** | 有限（需插件） | 天然支持（Offset） | 支持 | 有限 |
| **事务消息** | 不支持原生 | 支持（2PC） | 原生支持 | 支持 |
| **延迟消息** | 插件支持 | 时间轮实现 | 原生支持 | 支持 |
| **流处理** | 不支持 | Kafka Streams / ksqlDB | 有限 | 不支持 |
| **运维复杂度** | 低 | 高 | 中 | 低 |
| **语言绑定** | 几乎所有语言 | 几乎所有语言 | Java 为主 | 多语言 |
| **成熟度** | 成熟 | 成熟 | 成熟 | 较老 |
| **社区活跃度** | 非常活跃 | 非常活跃 | 活跃（国内为主） | 较低 |

### 8.1 选型建议

| 场景 | 推荐 |
|------|------|
| **通用任务队列、微服务解耦** | RabbitMQ |
| **高吞吐日志收集、事件流、数据管道** | Kafka |
| **金融级交易、低延迟、事务消息** | RocketMQ |
| **IoT、移动端、弱网环境** | MQTT 协议（EMQX / Mosquitto） |
| **已有 JMS 生态的 Java 项目** | ActiveMQ（建议迁移） |
| **云原生、Serverless** | 云服务商托管 MQ（SQS / RocketMQ 云版 / Kafka 云版） |

## 9. 最佳实践要点

### 9.1 消息设计
- 消息体不要太大（建议 < 1MB，Kafka 默认单条 < 1MB）
- 使用合适的序列化格式（JSON / Protobuf / Avro）
- 包含唯一消息 ID 用于去重和追踪
- 设计兼容的 Schema 版本策略（Schema Registry）

### 9.2 可靠性
- 生产端开启 Confirm 模式
- 重要消息开启持久化
- 消费端手动 Ack，处理完再确认
- 实现幂等消费（业务键 + 状态表 / Redis 去重）
- 配置死信队列兜底

### 9.3 性能
- 合理设置 Prefetch Count（RabbitMQ），避免消息堆积在消费者内存
- 批量发送/批量消费提升吞吐
- 压缩消息体（gzip / snappy / zstd）
- 监控队列积压，设置告警阈值

### 9.4 监控
- 队列深度（积压数）
- 消费延迟（Lag）
- 消费速率 / 生产速率
- 重试次数 / 死信数量
- 连接数 / 通道数

### 9.5 常见陷阱
| 陷阱 | 说明 | 对策 |
|------|------|------|
| **消息丢失** | 生产者未确认、消费者自动 Ack | 开启 Confirm + 手动 Ack |
| **重复消费** | 网络重试导致同一条消息多次投递 | 消费端幂等 |
| **消息堆积** | 消费速度跟不上生产速度 | 增加分区/消费者、优化消费逻辑 |
| **顺序错乱** | 多消费者并行消费、重试导致乱序 | 按分区/队列保证顺序，或用业务时间戳排序 |
| **死信堆积** | DLQ 无消费导致内存/磁盘满 | 配置 DLQ 监控和告警 |
| **连接泄漏** | 频繁创建/销毁连接 | 使用连接池 |
