# 登录注册设计

> 从注册到登录的完整流程设计，涵盖密码策略、验证码、邮箱/手机验证。

---

## 1. 注册流程

### 1.1 标准注册流程

```
用户填写表单
    ↓
验证输入（格式、密码强度）
    ↓
验证码校验（图形/短信/邮箱）
    ↓
检查用户是否已存在
    ↓
密码哈希处理
    ↓
存入数据库
    ↓
发送欢迎通知（可选）
    ↓
自动登录 / 跳转登录页
```

### 1.2 邮箱/手机号验证

```
注册提交
    ↓
发送验证码到邮箱/手机
    ↓
用户输入验证码
    ↓
验证码校验
    ↓
账号激活完成
```

**验证码存储：**
- 存 Redis，`key = verify:{邮箱/手机}`, `value = 验证码`
- 设置 TTL（通常 5 分钟）
- 限制发送频率（60 秒内不可重发）

### 1.3 注册信息校验规则

| 字段 | 规则 | 说明 |
|------|------|------|
| 用户名 | 3~20 位，字母数字下划线 | 不能纯数字 |
| 密码 | 8 位以上，含大小写+数字 | 见下方密码策略 |
| 邮箱 | 标准邮箱格式 | 可选 |
| 手机号 | 11 位数字，号段校验 | 可选 |

## 2. 密码策略

### 2.1 密码强度要求

```
❌ 弱密码: 123456, password, qwerty
✅ 中等:   Hello123
✅ 强:     H3ll0_W0rld!  (大小写+数字+符号+8位以上)
```

**推荐规则：**
- 最少 8 位（建议 12+）
- 至少包含：大写字母、小写字母、数字
- 符号可选但推荐
- 禁止常见密码（`password`、`123456`、`qwerty`）
- 禁止与用户名相同

### 2.2 密码存储

```
❌ 明文存储          ← 绝对禁止
❌ MD5/SHA-1 直接哈希  ← 彩虹表秒破
❌ MD5/SHA-1 + 固定 salt  ← 不够安全
✅ bcrypt/scrypt/argon2  ← 推荐
```

```python
# ✅ bcrypt 示例
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

# 验证
bcrypt.checkpw(input_password.encode(), stored_hash)
```

```java
// ✅ Spring Security BCrypt
PasswordEncoder encoder = new BCryptPasswordEncoder(12);
String hash = encoder.encode(password);
boolean match = encoder.matches(rawPassword, hash);
```

### 2.3 密码重置

```
用户请求重置密码
    ↓
发送重置链接（带 Token）到注册邮箱/手机
    ↓
用户点击链接
    ↓
Token 校验（未过期、未使用）
    ↓
输入新密码
    ↓
更新密码 + 使旧 Token 失效
    ↓
通知用户密码已更改
```

**重置 Token 设计：**
- 使用 UUID 或 JWT
- 存入 Redis / 数据库
- 有效期通常 15~30 分钟
- 一次性使用（使用后立即失效）
- 重置后使所有当前登录 Session 失效（可选）

## 3. 登录流程

### 3.1 标准登录流程

```
用户输入账号密码
    ↓
校验验证码（有的话）
    ↓
查询用户（账号/邮箱/手机）
    ↓
用户不存在 → 返回"账号或密码错误"
    ↓（统一错误提示，不透露是账号错还是密码错）
密码校验
    ↓
密码错误 → 记录失败次数
    ↓
账号锁定? → 返回"账号已锁定"
    ↓
登录成功 → 更新最后登录时间/IP
    ↓
生成 Token / Session
    ↓
返回给客户端
```

### 3.2 统一错误提示

**不要区分**"用户不存在"和"密码错误"，统一返回：

```
❌ "用户不存在"     ← 攻击者可据此枚举有效账号
❌ "密码错误"       ← 确认了账号存在
✅ "账号或密码错误"  ← 安全
```

### 3.3 登录安全措施

| 措施 | 说明 |
|------|------|
| **验证码** | 连续失败 3~5 次后要求输入 |
| **账号锁定** | 连续失败 N 次后锁定 N 分钟 |
| **设备指纹** | 记录常用设备，异常设备需验证 |
| **异地登录检测** | 检测 IP 归属地变化 |
| **登录通知** | 异常登录发送短信/邮件提醒 |
| **并发登录限制** | 可配置单设备/多设备登录 |

## 4. 验证码

### 4.1 验证码类型

| 类型 | 安全性 | 用户体验 | 适用场景 |
|------|--------|----------|----------|
| 图形验证码 | 低 | 差 | 防简单脚本 |
| 短信验证码 | 高 | 中等 | 注册、登录、重置密码 |
| 邮箱验证码 | 中 | 中等 | 注册验证 |
| 滑动验证码 | 高 | 好 | 登录、敏感操作 |
| 行为验证（无感） | 高 | 最好 | 后台风控 |

### 4.2 验证码设计要点

```
短信验证码:
  - 6 位数字（平衡安全与输入便利）
  - 有效期 5 分钟
  - 60 秒内不可重发
  - 每日上限 10 次
  - 校验后立即失效（防止重放）

图形验证码:
  - 4~6 位字母数字
  - 干扰线、噪点、扭曲
  - 有效期 2 分钟
  - 服务端验证（不信任前端）
```

## 5. 登录状态管理

### 5.1 登录状态信息

```json
{
  "userId": 1001,
  "username": "alice",
  "role": "admin",
  "loginTime": "2025-01-01T10:00:00Z",
  "lastIp": "192.168.1.1",
  "tokenVersion": 2
}
```

**不要存入 Session/Token 的敏感信息：**
- ❌ 密码哈希
- ❌ 手机号（非必要）
- ❌ 支付信息

### 5.2 退出登录

```
用户点击退出
    ↓
服务端处理:
  Session 方案 → 删除 Session 数据
  JWT 方案     → 加入黑名单 / 递增 tokenVersion
    ↓
客户端处理:
  清除本地 Token
  跳转登录页
```

### 5.3 Token 刷新与失效

见 [Session 与 JWT](./session-jwt.md) 的 Token 刷新部分。

## 6. 多设备登录策略

| 策略 | 说明 | 适用 |
|------|------|------|
| **单设备登录** | 新登录使旧设备 Token 失效 | 金融、敏感系统 |
| **多设备共存** | 所有设备保持在线 | 通用 |
| **限制设备数** | 最多 N 台设备同时在线 | 视频会员 |

**实现方式：**
- 每个登录生成唯一的 Token/Session
- 服务端记录每个用户的活跃 Token 列表
- 单设备策略：新登录 → 旧 Token 加入黑名单
- 限制设备数：超出时踢掉最早登录的设备
